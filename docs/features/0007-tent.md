# tent — Harness Containment as the Policy Boundary

## Abstract

This record proposes **tent**: a clown-managed container/sandbox that
wraps the launched provider (claude, codex, crush, opencode, circus,
or future harnesses) and enforces clown's policy by controlling the
harness's view of the world — filesystem, network, command surface,
and lifecycle — rather than by configuring the harness itself.

Status is **testing**. The tracer-bullet shape (claude-only,
podman-on-linux, podman-machine-on-darwin, --tent opt-in) has been
proven end-to-end on both platforms — both `--version` and a real
plugin-tool MCP round-trip through gvproxy work without errors.
The framing is committed. Outstanding (gating promotion to
`accepted`):

- Making tent the *default* execution path (not opt-in). FDR-0009's
  `--naked` is the named escape. Issue #62 (claude-code 2.1.123+
  bump) is the bridge between tent-as-opt-in and tent-as-default.
- Codex/crush/opencode/circus support (currently claude-only).
- A documented loopback-reachability story across platforms
  (linux native uses `--network=host` + same-host loopback;
  darwin uses `--network=host` + URL rewriting to
  `host.containers.internal`).

## Motivation

Clown's load-bearing invariant is "settings are unbypassable" —
managed-settings deny destructive Bash patterns, disable bypass-
permissions mode, install PreToolUse hooks for permission decisions,
and override commit/PR attribution. These properties must hold even
when the user types the harness's own escape-hatch flags.

Today the invariant is enforced *inside* the claude-code binary
through a patched cli.js that redirects `/etc/claude-code` to a
managed-settings file under the nix store. The path is hardcoded
in the binary, so the user can't reach over it without replacing the
binary. That's the unbypassability we depend on.

A cross-harness investigation surfaced that no other harness clown
wraps offers a comparable mechanism on Linux:

- **Claude Code** has the unbypassable Linux story (cli.js patch +
  `/etc/claude-code`), but upstream switched to a Bun-compiled
  binary distribution at 2.1.121+. There is no `cli.js` to patch
  and binary string substitution is length-constrained — our
  replacement is longer than the slot. Bumping past 2.1.111 is
  blocked on this. See [#62][i62].
- **Crush** has clean declarative permission config and PreToolUse
  hooks, but it's all user-config: bypassable by running the crush
  binary directly with a different config file.
- **OpenCode** has the most expressive permission language (three-
  state `allow`/`deny`/`ask` with wildcards), but its `SECURITY.md`
  explicitly disclaims being a security boundary and recommends
  Docker/VM for true isolation. A real managed-settings tier
  exists on macOS via Apple MDM `.plist`, but there is no Linux
  analog.

The conclusion: stop trying to enforce policy *inside* the harness;
enforce it in the *environment the harness sees*. Tent is that
environment.

[i62]: https://github.com/amarbel-llc/clown/issues/62

## What tent is

Tent is the layer that sits between clown and the launched provider.
Conceptually:

```
clown
  └── tent (controlled environment)
        └── provider (claude/codex/crush/opencode/circus)
              └── ...child processes the provider spawns
```

Tent's job is to expose a *policy-conforming view* of the host to
whatever runs inside it:

- **Filesystem.** A bind-mounted subset of the user's working tree
  (writable), `/nix/store` (read-only, only the closure paths the
  provider needs), and tmpfs everywhere else. Sensitive paths
  (`~/.ssh`, `~/.gnupg`, `/etc` outside what the provider needs)
  are not visible.
- **Network.** Allowlisted egress (e.g., `api.anthropic.com`,
  `api.openai.com` for codex, the local llama-server's loopback
  for circus), denied otherwise. Loopback inside the namespace is
  available to providers that need it (circus, plugin-host).
- **Command surface.** `bash`, `git`, the provider's own tooling,
  and clown-installed hook helpers — but not `sudo`, `su`,
  `mount`, or any other capability-elevating command.
- **Lifecycle.** Tent owns the provider's process tree. When clown
  exits or the user signals, tent tears down the namespace cleanly
  and there is no surviving state outside the bind-mounted working
  tree.

Crucially, tent's policy is set by clown at launch time, not by the
provider's own config. A user passing `--dangerously-skip-permissions`
to claude inside tent still cannot `rm -rf /` because tent has no
`/` to remove from — only the bind-mounted view it was given.

## Existing pieces tent ties together

- **`clownbox`** is currently disabled in the build (`flake.nix:579`,
  `clownboxCliPath = ""`) but was a fork of `numtide/claudebox`
  using bubblewrap to confine claude. Exact precedent for the
  pattern, just claude-only and with bubblewrap's Linux-only
  posture. Tent generalizes from one harness to many and revisits
  the sandbox-tech question.
- **ADR-0005** ("Nix Builder Sandbox as the Confinement Layer")
  established the philosophy that the sandbox boundary is the
  permission boundary. Tent is the runtime analog of that idea —
  same property, different sandbox layer.
- **ADR-0006** ("clown-stdio-bridge Trusts Loopback Clients")
  leans on the namespace boundary for trust. Tent is where that
  namespace lives at runtime.
- **RFC-0006** ("clown-plugin sandboxing posture", open) — tent's
  shape determines what plugins can see and reach.
- **RFC-0007** ("Sandbox-escaping bridge for hooks", open) — hooks
  fired by the provider need to reach back through tent to clown
  for permission decisions. Tent's bridge mechanism is what RFC-0007
  specifies.
- **FDR-0001** ("Sandboxed Subagents", draft) — applies the same
  philosophy to *subagent* processes inside a clown invocation;
  tent applies it to the *outer* provider container. Adjacent but
  not redundant.
- **FDR-0009** ("Naked Emergency Bypass") — establishes a flag
  posture for "no managed-settings, raw provider" mode. Tent
  inherits the same vocabulary: a `--no-tent` or equivalent flag
  for emergencies.
- **Fence** (Use-Tusk) and **bubblewrap** are candidate tents.
  Fence's loopback-isolation behavior was characterized in the
  tracer-bullet experiment recorded under ADR-0007's exploration.

## Open architectural questions

These are the load-bearing decisions that gate moving to `proposed`.
Each likely warrants its own ADR.

1. **Which sandbox tech?**
   - **bubblewrap** — clownbox precedent, well-trodden, Linux-only.
   - **fence** (Use-Tusk) — declarative seatbelt+landlock, supports
     macOS and Linux, smaller community. Investigated this session
     and it isolates network namespaces correctly.
   - **podman/docker** — heavier, standardized, well-understood
     ops story, but a much larger closure for end users.
   - **unshare(1) primitives directly** — minimal closure, full
     control, but everything else is hand-rolled.

2. **What does tent expose to the harness?**
   - Filesystem: which subset of `$HOME`, which `/nix/store` paths,
     which writable scratch?
   - Network: domain allowlist; clown-managed proxy with TLS
     termination for audit; or just an L3/L4 allowlist via
     iptables/netns?
   - Command policy: blanket deny + clown-allowlist, or default-allow
     with deny-known-bad?

3. **How do plugins/hooks reach back through tent?**
   This is RFC-0007's territory. clown-plugin-host runs *outside*
   tent (so it can be the policy-decision authority); plugin MCP
   servers may run inside or outside; tent provides a bridge socket
   for hooks to invoke clown's deciders. The exact shape is
   undecided.

4. **Per-provider profiles vs single tent shape?**
   Different providers have different needs:
   - claude needs anthropic.com egress, no local loopback target.
   - codex needs openai.com egress.
   - circus needs loopback to the user's `llama-server` daemon —
     and that daemon runs *outside* tent, so tent must bridge.
   - crush/opencode need OpenAI-compat or gateway egress.
   The question is whether tent has one parametric profile or N
   per-provider tent images. FDR-0006 (single-entrypoint matrix)
   is the upstream framing.

5. **Relationship to clown-plugin-protocol.**
   Plugins are MCP servers spawned by clown-plugin-host today (they
   listen on loopback; the patched plugin manifest points the
   harness at the loopback URL). Inside tent, the harness's loopback
   is the *namespace*'s loopback — not the host's. Either plugin-
   host runs inside the namespace, or there's a socat/proxy bridge.
   See RFC-0006.

6. **Naked vs tented mode.**
   FDR-0009 (naked emergency bypass) already establishes that clown
   supports a "naked" mode for emergency bypass. Tent should be the
   default for every provider, with `--no-tent` (or equivalent) as
   the named opt-out. Implementation: same flag posture as today's
   `--disable-clown-protocol`.

7. **macOS posture.**
   As of 2026-05-11, tent ships on darwin via podman-machine —
   `clown --tent` runs a linux container on the user's mac through
   an Apple Virtualization Framework VM, and gvproxy handles the
   loopback story for plugin-host (see issue #70 and FDR-0007's
   2026-05-11 Update entries). The platform layer is recorded in
   `amarbel-llc/eng FDR-0003 — Podman as a Darwin Platform Service`.

   The remaining open question on macOS posture is whether
   *another* sandbox tech (Fence-on-Seatbelt, nix-build sandboxing
   per ADR-0005) would give a uniform Linux+macOS posture with
   fewer moving parts than "podman on linux, podman-machine on
   darwin." The status quo works, but:

   - It costs a long-lived VM per darwin host (memory, disk,
     login-time bootstrap).
   - The plugin-host topology requires a host-replacement hack
     (`URLHostRewrite = "host.containers.internal"`) that doesn't
     exist on linux.
   - The unpatched claude-code path requires cross-building from
     darwin via nix-darwin's `linux-builder` module.

   A future ADR may revisit Fence-on-Seatbelt as a darwin-native
   alternative, but it's no longer load-bearing for tent's v0
   shape.

8. **Build-time vs runtime policy injection.**
   Today, claude-code's managed-settings are baked into the patched
   binary at *build* time. Tent shifts this to *launch* time:
   clown computes the policy from the user's flags + builtin
   defaults, then materializes it into the tent's view. This is
   strictly more flexible (per-invocation policy variation, no
   rebuild for new policy) but trades the binary-level guarantee
   for a runtime-construction guarantee. Whether that trade-off is
   acceptable for the unbypassability invariant is the central
   open question.

## Non-goals

- **Tent is not a security sandbox against an actively malicious
  harness.** The threat model is a cooperative harness that may
  make mistakes or accept risky tool calls. A harness actively
  trying to break out of the namespace is out of scope; the right
  substrate for that would be Firecracker or gVisor.

- **Tent does not replace the harness's own permission model.**
  Inside tent, the harness's permission prompts, hooks, and
  config still apply. Tent is a *defense in depth* layer — the
  outermost ring. The harness's own policy continues to be the
  inner ring.

- **Tent does not host plugins by itself.** Plugins are clown's
  responsibility, run by clown-plugin-host, and reach the harness
  via the existing clown-plugin-protocol. Tent's role is purely
  to confine the harness; plugins remain a parallel concern.

- **Tent does not implement audit logging in v1.** Logging the
  harness's tool calls, network requests, and filesystem accesses
  is a natural extension and is mentioned in adjacent RFCs, but
  is out of scope for the initial cut.

## Tracer-bullet status (2026-05-10)

A first cut has landed under `--tent`, deliberately scoped to prove
the wrapping mechanism end-to-end rather than to answer the gating
ADR questions. Committed:

- **Sandbox tech**: podman (`pkgs.podman` linux-only). Image built
  via `dockerTools.buildLayeredImage`; `/nix/store` is bind-mounted
  read-only so the claude binary is referenced by its store path
  rather than baked into the image.
- **Coverage**: claude only. Codex/crush/opencode/circus paths
  unchanged.
- **Activation**: opt-in. `--tent` (or `CLOWN_TENT=1`) wraps; the
  un-tented path is the default and is byte-for-byte unchanged.
- **Network**: `--network=host` so plugin-host loopback URLs
  continue to work without bridging.
- **Working surface**: `$PWD` writable, `/nix/store` ro,
  `~/.claude` and `~/.config/claude` writable, `$TMPDIR` writable
  (for plugin-host's staged manifest dirs).

Deferred to follow-ups: egress allowlisting, per-provider profiles,
codex/crush/opencode/circus support, hook-bridge integration
(RFC-0007), audit logging. (macOS posture is no longer deferred —
see §7 above; tent ships on podman-machine on darwin as of
2026-05-11.) The bypassability story
(#62, claude-code 2.1.123+ bump) is *unblocked but not closed* by
this cut — making tent the default and removing the binary-patch
load on managed-settings remain follow-ups.

Code pointers: `cmd/clown/main.go` (flag + `tentExecutor`),
`internal/tent/tent.go` (argv builder), `internal/buildcfg`
(`PodmanPath`, `TentImageRef`, `TentImageTarball`), `flake.nix`
(`tentImage`, ldflags).

### Update (2026-05-11): unpatched claude-code inside tent

The tracer-bullet's first cut ran the *same* managed-settings-patched
claude-code 2.1.111 inside tent that runs outside, which was redundant —
tent is the policy boundary, so the inner ring's managed-settings shim is
defense-in-depth that just adds binary patching cost without adding
guarantees.

Tent now uses an unpatched claude-code sourced from
`numtide/llm-agents.nix` (a flake input we already had for crush; it
ships claude-code as a self-contained binary fetched from Anthropic's
GCS release manifest, currently version 2.1.138). The patched 2.1.111
remains the default for un-tented clown until issue #62 is resolved
(cli.js substitution needs to be ported to binary-string substitution).

Two corresponding changes inside tent:
- No managed-settings JSON shipped — the inner claude is upstream-stock.
- Clown's provider-level safety denylist (`Bash(*)`, `Agent(Explore)`,
  `WebFetch`) is no longer applied. Tent IS the boundary.

claude's own default permission prompts still run inside the container
(`--dangerously-skip-permissions` is not set). The user wanted
"permissive but not bypass-all" — tent absorbs the outer ring, but the
inner harness still asks before consequential tool calls.

`--select` to pick a specific claude-code version at launch is filed as a
follow-up (issue #65). The trade-off between build-time-baked versions
and runtime-fetched arbitrary versions is recorded there.

Code pointers (additional): `internal/buildcfg.ClaudeTentCliPath`
(unpatched binary path baked at build time), `cmd/clown/main.go`'s
`resolveClaudeForRun` (dispatch helper), `flake.nix:tentClaudeCliPath`
(`pkgs-llm-agents.claude-code`).

### Update (2026-05-11, cont'd): operational fit-out and follow-ups

End-to-end testing on Ubuntu 24.04 surfaced a set of operational
gotchas. Each was patched in this session; recording them here so a
future port to a different host distro or to a different sandbox tech
doesn't re-learn them.

- **Rootless podman prereqs.** `--userns=keep-id` requires
  `newuidmap`/`newgidmap` (setuid helpers from the `uidmap` /
  `shadow-utils` package) and a `/etc/subuid` entry for the user.
  Default Ubuntu 24.04 has the subuid line but not the uidmap
  binaries; the failure is a podman-side "command required for
  rootless mode with multiple IDs" that doesn't point at the fix.
  Clown now preflights both in `newTentExecutor` and surfaces an
  actionable install hint (`sudo apt install -y uidmap`). Filed
  amarbel-llc/eng#73 to wire this up reproducibly via home-manager.

- **Locale pinning.** Host `LC_ALL=en_US.UTF-8` leaking into the
  container produced a `setlocale: cannot change locale` warning on
  every spawned bash (claude shells out frequently). Resolved by
  pinning the image's env to `LC_ALL=C.UTF-8` / `LANG=C.UTF-8`
  (built into glibc 2.35+; no archive needed) and dropping
  `LC_ALL`/`LANG` from `DefaultEnvPassthrough`.

- **Sibling-file mounts.** claude-code stores its top-level user
  config at `~/.claude.json` (sibling of `~/.claude/`, not a child).
  Our first mount list only covered `~/.claude/` + `~/.config/claude/`,
  so claude inside the tent reported "Claude configuration file not
  found". Fix: explicitly add `~/.claude.json` as a writable bind
  mount, and pre-initialize it to `{}` on the host if missing so
  podman doesn't auto-create the bind source as a directory.

- **Mount precedence is parent-loses, child-wins.** Empirically
  verified via `zz-explore/tent-mount-precedence` recipe: when the
  same path tree is bind-mounted twice (e.g. `$TMPDIR` writable +
  plugin-host's staged dirs inside `$TMPDIR` as `:ro`), podman
  honors the inner mount. No deduplication needed in the tent code.

- **Image-load UX.** First-time `podman load -i <tarball>` takes
  ~5-30s and previously dumped raw "Copying blob …" lines to stderr.
  Replaced with a bubbletea spinner + rolling 5-line tail
  (`cmd/clown/tent_loader.go`), collapsing to a single
  "✓ Tent image cached" line on success. Non-TTY callers (CI,
  redirected output) keep streaming raw output to stderr.

- **Run-phase silence.** `podman run -t` takes over the TTY directly,
  so we cannot run a real spinner during claude's startup (CLAUDE.md
  auto-discovery, plugin sync, keychain reads — 5-30s before first
  paint). A static `Starting claude inside tent…` stderr line prints
  before exec; claude's TUI paints over it. A real PTY-proxy spinner
  is filed as amarbel-llc/clown#67.

- **`--bare` discovered upstream.** claude-code 2.1.13x+ ships a
  `--bare` flag that natively skips hooks/LSP/plugin-sync/auto-memory/
  keychain reads/CLAUDE.md-auto-discovery and forces explicit auth.
  That's a sizable chunk of what clown's managed-settings shim was
  doing. Worth exploring as a replacement for the cli.js patch (which
  blocks bumping the un-tented claude past 2.1.111; see #62). Filed
  as amarbel-llc/clown#66.

- **`--select` version picker.** Tent uses whatever the
  `numtide/llm-agents.nix` input is currently pinned to (latest at
  build time). Picking a specific older version at launch — for
  downgrade after a bad upstream release, A/B comparison, or
  pinning-for-the-session — would need either a baked menu of N
  versions, a runtime nix-build against historical refs, or a hybrid.
  Trade-off recorded at amarbel-llc/clown#65.

Open issues from this iteration (all open for triage):
- amarbel-llc/clown#65 — `--tent --select` version picker
- amarbel-llc/clown#66 — explore `--bare` vs the managed-settings patch
- amarbel-llc/clown#67 — PTY-proxy spinner for tent's run phase
- amarbel-llc/eng#73 — install podman + uidmap reproducibly

### Update (2026-05-11, cont'd): partial darwin enablement

`flake.nix` previously gated *all* tent build-config under a single
`tentEnabled = pkgs.stdenv.isLinux` predicate, so a darwin build left
`ClaudeTentCliPath` empty and the runtime bailed with a misleading
"(linux-only)" message before any podman code ran. With an external
podman-machine VM (see amarbel-llc/eng zz-pocs/podman-darwin), the
remaining linux-isms turn out to be small.

This change splits the predicate:

- `tentImageEnabled = pkgs.stdenv.isLinux` continues to gate the
  `dockerTools.buildLayeredImage`-based `tentImage`, `tentImageRef`,
  `tentImageTarball`, and the nixpkgs `pkgs.podman` binary — none of
  which build or evaluate on a darwin builder.
- `tentClaudeEnabled = true` always wires `ClaudeTentCliPath` from
  `pkgs-llm-agents.claude-code` (a fetchurl + binary install that
  ships a darwin variant).

The runtime change is matched on the cmd/clown side:

- `resolveClaudeForRun`'s error message drops the "(linux-only)"
  phrasing and now reflects the actual cause (build didn't pass
  the ldflag) rather than a fictional platform restriction.
- `preflightUserNs` short-circuits on non-linux: `newuidmap` and
  `/etc/subuid` only matter for native rootless podman; on darwin
  the user-namespace mapping lives inside the podman-machine VM.

The smoke test from issue #68 (`clown --tent -- --version`) now
clears the `ClaudeTentCliPath` check. It still trips on
`PodmanPath`/`TentImageRef` emptiness — those remain linux-only at
build time. Wiring those for darwin (runtime PATH lookup for podman,
plus an image-distribution strategy that doesn't depend on a darwin
builder) is the next step toward full darwin tent support.

follow-up TBD for the podman + image plumbing.

### Update (2026-05-11, cont'd): podman + image plumbing on darwin

The follow-up that #68's update flagged (PodmanPath / TentImageRef
emptiness, image-distribution strategy) is now in. Three small
findings reshaped the plan from #69's original sketch:

- **`pkgs.podman` builds on darwin.** The constraint #69's body
  claimed (`pkgs.podman` unsupported on darwin) was wrong — clown's
  pinned nixpkgs ships `aarch64-darwin.podman` 5.8.2 as a normal
  derivation. On darwin the binary is a thin client that proxies
  to a podman-machine VM. So we bake it unconditionally now:
  `tentPodmanEnabled = true`.
- **`dockerTools.buildLayeredImage` builds on darwin but produces
  a defective image.** Tested directly: it evaluates, produces a
  tarball whose manifest claims `linux/arm64` while wrapping mach-O
  binaries. The fix is to produce the image with `system =
  "aarch64-linux"`, which on a darwin host needs a remote linux
  builder. nix-darwin's `nix.linux-builder` module spins one up via
  Apple Virtualization Framework. With that enabled,
  `packages.aarch64-linux.tent-image` builds cleanly. The image is
  not baked into the darwin clown binary (no `TentImageTarball`);
  the eng-side POC's `phase4-load-image` recipe does the cross-build
  and `podman load` explicitly.
- **`TentImageRef` is just a string tag.** Baking it on darwin
  costs nothing and makes the runtime's image-store miss surface
  as a clean "image not present locally and no tarball is wired
  in" instead of an opaque earlier error.

The flake now:
- Bakes `tentPodmanPath` on every platform (`tentPodmanEnabled`).
- Bakes `tentImageRef` on every platform (string tag).
- Leaves `tentImageTarball` empty on darwin (image must be
  pre-loaded by the user via the eng POC).
- Exposes `packages.aarch64-linux.tent-image` as a named output so
  cross-builds from darwin via linux-builder work.

The smoke test from #68 now reaches the image-store miss instead of
the empty-build-config gates:

    $ clown --tent -- --version
    clown: tent image clown-tent:0.2.21 not present locally and
    no tarball is wired in

After `just phase4-load-image` in the eng POC, that should clear too.
End-to-end smoke test results land in FDR-0003 on the eng side once
phase 4 runs.

### Update (2026-05-11, cont'd): end-to-end smoke test green on darwin

Driving the eng-side POC (zz-pocs/podman-darwin) phase 4 surfaced
two more darwin-specific gaps that the previous updates had missed.
Both are fixed; `clown --tent -- --version` now prints
`2.1.138 (Claude Code)` on a podman-machine-backed darwin host.

The two gaps:

1. **Bind-source directories don't exist on a fresh darwin host.**
   tent.BuildArgs unconditionally bind-mounts `~/.claude/`,
   `~/.config/claude/`, and `~/.claude.json`. Only the last has
   an `ensure*` preflight today. On a darwin host where claude has
   never run, `~/.config/claude/` simply doesn't exist (claude on
   darwin only writes to `~/.claude/` post-first-run, and XDG
   `~/.config/claude/` may never exist at all). podman refuses
   to start with `statfs … no such file or directory`. Fix:
   added `ensureClaudeBindSources` to mkdir -p the bind dirs
   with 0o700 when missing, mirroring `ensureClaudeJSON`'s
   pattern. Three tests cover create-missing, leave-existing,
   reject-file-at-dir.

2. **`ClaudeTentCliPath` resolved to the wrong-platform binary.**
   The earlier update wired `tentClaudeEnabled = true` on darwin,
   sourcing the binary from `llm-agents.packages.<system>
   .claude-code`. On darwin `<system>` is `aarch64-darwin`, which
   gave us a mach-O wrapper script — exec'd inside the linux
   container, it failed with `Exec format error`. The binary
   baked here runs *inside* the container, so it MUST be the
   linux variant regardless of host. Fix: map every system to its
   linux counterpart (`aarch64-darwin → aarch64-linux`,
   `x86_64-darwin → x86_64-linux`, linux → identity), and source
   `tentClaudeCliPath` from `llm-agents.packages.<linux-system>
   .claude-code`. On darwin this requires nix-darwin's linux-builder
   to produce the closure (see eng-side FDR-0003 for the host setup).

End-to-end smoke test on darwin (with podman-machine VM up,
`clown-tent:0.2.21` loaded via the eng POC's `phase4-load-image`):

    $ clown --tent -- --version
    Starting claude inside tent…
    2.1.138 (Claude Code)

Closes amarbel-llc/clown#69.

### Update (2026-05-11, cont'd): plugin-host loopback on darwin

`--version` doesn't exercise the plugin pipeline, but any plugin
flow under `clown --tent` on darwin would have broken silently —
plugin servers bind to `127.0.0.1:<port>` on the mac, and under
`--network=host` the container's `127.0.0.1` is the
podman-machine VM's loopback, not the mac's. claude inside the
tent could not reach any of clown's plugin servers.

The eng-side POC's `phase4-net-*` probes (zz-pocs/podman-darwin
justfile) verified the gvproxy reachability matrix:

  - `--network=host`, container's 127.0.0.1 → mac:      UNREACHABLE
  - `--network=host`, host.containers.internal → mac:   REACHED
  - default network, host.containers.internal → mac:    REACHED
  - default network, host.docker.internal → mac:        REACHED

So gvproxy injects `host.containers.internal` *unconditionally*
on podman-machine — `--network=host` is fine; only the host
portion of the URL needs to change.

Fix: add a `URLHostRewrite` field to `pluginhost.Host` and a
sibling `Handshake.URLWithHostRewrite` that swaps the host
portion of a server URL while preserving port and path. The
field is empty by default (linux native: no rewrite). cmd/clown
sets it to `host.containers.internal` when tent is active on
darwin via a new `pluginURLHostFor` helper.

Critically, the rewrite is applied *only when emitting the URL
into the compiled plugin manifest* — clown's own healthcheck
dial and shutdown loop keep using the unmodified address
because they live in the same process as the bind. This avoids
turning a loopback dial into a remote one.

Tests: parameterized table covering empty-override, ipv4,
ipv6 (`[::1]`), sse-vs-streamable, and malformed-address
fallback. Plus a `pluginURLHostForGOOS` table covering the
five combinations of (tent, goos).

Closes amarbel-llc/clown#70.

### Update (2026-05-12): host-devshell passthrough (interim)

`DefaultEnvPassthrough` deliberately strips `PATH` (and adjacent
path-list vars like `PKG_CONFIG_PATH`, `XDG_DATA_DIRS`) to keep the
container's tool surface predictable — the agent inside tent sees a
known set of binaries, not a copy of whatever the user's shell happened
to populate. A consequence: when the user runs `clown --tent` from
inside a nix devshell (direnv-loaded or `nix develop`-entered), none of
the devshell's tools are on the in-container `PATH`, even though every
one of them lives under `/nix/store` and is already visible via the
read-only `/nix/store` bind mount.

The principled fix is the planned `--profile` system, which will own
named (provider, backend, model, environment) tuples and can describe
which host env each profile passes through. Profiles are ~weeks out;
until then, this update adds an opt-in interim:

- `--tent-pass-devshell` (and matching tent config field) sets the
  container's `PATH` to the host `$PATH` filtered to its
  `/nix/store/<hash>-<name>/bin` entries, passed via
  `--env PATH=<filtered>` (explicit value, not value-by-reference, so
  the container never sees host entries it can't resolve).
- The tent image bakes no baseline `PATH` today, so this is a clean
  replacement, not an append. If a future image grows a real baseline
  `PATH` (e.g. a bundled coreutils/bash floor so tent is usable without
  a devshell), revisit this as "prepend filtered devshell entries to
  the image baseline" rather than full replace.
- Off by default. Auto-detection from `IN_NIX_SHELL` was considered
  and rejected — it conflates "user happens to be in a devshell" with
  "user wants the devshell visible to the agent", and the profile
  system will subsume the knob anyway.
- No new bind mounts: every forwarded entry already resolves through
  the existing `/nix/store` ro mount, on both linux native and darwin
  (podman-machine mounts the host `/nix/store` into the VM, and tent
  bind-mounts `/nix/store` into the container from there).
- The flag name telegraphs its interim status. Profiles will own the
  durable interface; this flag is fair game to remove once they ship.

Deliberately deferred (to profiles, not to v0):

- The same `/nix/store`-filter rule applied to `PKG_CONFIG_PATH`,
  `XDG_DATA_DIRS`, and similar path-list vars devshells populate.
  Mechanism is identical; v0 is `PATH`-only to keep the change small.
- Non-`/nix/store` passthrough (e.g. `~/.local/bin`). Would require
  new bind mounts, which is a different conversation and belongs in
  the profile schema.
- Per-binary allowlisting. Probably never lands — the agent already
  sees the whole devshell via the store mount, so filtering at the
  `PATH` grain is theater rather than containment.

Tracking: amarbel-llc/clown#73.

### Update (2026-05-19): tool-surface shape (allowlist + agent-socket forwarding)

The interim `--tent-pass-devshell` flag (#73 above) only covers PATH —
it leaves the question of *what host state the agent inside the tent
can see* untouched. End-to-end probes against `clown-tent:0.3.0` (via
the new `zz-explore/tent-probe` recipe) surfaced what's needed to make
a `clown --tent` session actually usable for development work without
opening the boundary FDR-0007 sets up.

Findings from the probe loop:

- The image baseline (`bashInteractive + coreutils + cacert +
  iana-etc`) does NOT need to grow. Every dev tool the agent needs is
  already in the user's home-manager profile, reachable through
  `/nix/store`.
- `nix` works inside the tent with three bind mounts: `/nix/var:ro`
  (daemon socket + profile-link targets), `/etc/nix:ro` (daemon
  config), and `$HOME/.nix-profile:ro` (the profile-link itself).
  `nix store info` against the host daemon succeeds (Trusted=0 — no
  cache push, but local builds work). No daemon-bridge socat
  acrobatics required.
- The current `FilterPathToNixStore` is too strict: it accepts
  `/nix/store/<hash>-<name>/bin` but rejects `~/.nix-profile/bin` and
  `/nix/var/nix/profiles/default/bin` because those are profile-link
  directories. Both resolve into `/nix/store` and are reachable
  through the existing bind — the filter just needs to follow
  symlinks.

The chosen shape — `C + F` from the design conversation:

- **Allowlist of host paths** bind-mounted read-only (the FDR vision):
  `~/.gitconfig`, `~/.config/git/`, `~/.config/nix/`,
  `~/.nix-profile/`, plus today's claude-specific paths. Fails-closed:
  a missing entry breaks a tool loudly rather than leaks a path.
- **Agent-socket forwarding** for secret-bearing operations: pass
  `$SSH_AUTH_SOCK` through (bind-mount the socket path, forward the
  env var). Combined with pivy-agent (which signs both SSH auth *and*
  git commits via one socket), this gives the in-tent agent `git push`
  + `git commit -S` capability without exposing any key material.
  `SSH_AUTH_SOCK` must be added to `DefaultEnvPassthrough`.
- **Relaxed PATH filter**: keep any entry whose `realpath` starts with
  `/nix/store/`. Admits home-manager and system profile-link
  directories without giving up the no-stale-host-PATH guarantee.
- **`IN_NIX_SHELL` auto-on**: reverse the 2026-05-12 explicit
  rejection. With the relaxed filter, the home-manager surface comes
  along for free even when the user isn't inside a devshell, so the
  knob really just controls the *devshell* layer. Auto-on when
  `IN_NIX_SHELL` is set.

Explicit non-goals for this cut:

- **No** `--tent-pass-profile` flag. The relaxed filter subsumes it.
- **No** image-baseline expansion. The image stays at bash + coreutils
  + cacert + iana-etc; tools come from the user's profile.
- **No** general `$HOME` bind mount. Allowlist only.

Follow-up: **synthesize a tent-specific git config** rather than
bind-mounting the user's. Today's C+F shape exposes `~/.gitconfig` and
`~/.config/git/` read-only, which is enough to make commits work but
means the in-tent agent inherits the host user's full git identity,
signing key, aliases, and arbitrary config (including any pager,
editor, or hook configs that don't apply inside a sandbox). A more
principled shape is for clown to *generate* a `.gitconfig` at tent
launch time — fixed identity (or derived from the user's, minus
sections clown wants to control), no host hook scripts, no pager that
won't paginate in a non-TTY context, and (most importantly) clown
controls the `user.signingkey` / `gpg.format` so commit signing flows
through the forwarded SSH agent without surprise. Filed as a
follow-up; not load-bearing for the C+F tracer-bullet but a known
direction.

Tool-surface probe recipe: `zz-explore/tent-probe CMD …` — runs
arbitrary scripts inside the tent image with knobs for userns mode,
extra mounts, and devshell PATH passthrough. Kept around as a
debugging loop for future tent shape changes.

### Update (2026-05-19, cont'd): PATH construction strategy

Companion to the C+F shape above: once we add the profile-link bind
mounts, the question is how to *construct* the in-tent PATH from the
host's. Today's `FilterPathToNixStore` keeps `/nix/store/<hash>/bin`
entries and drops everything else; the new evidence is that the
agent's most-wanted tools (`git`, `nix`, `jq`, `curl`) live behind
profile-link directories (`~/.nix-profile/bin`,
`/nix/var/nix/profiles/default/bin`) which the current filter drops.

Alternatives considered (debugged against `clown-tent:0.3.0` via
`zz-explore/tent-probe`):

| Strategy                                           | Hermetic        | Debuggable           | Extra mounts beyond Q1   | Tool-surprise risk |
| :--                                                | :--             | :--                  | :--                      | :--                |
| A. Keep filter as-is + bind profile dirs           | No ($HOME dep)  | Yes                  | none (Q1 covers)         | low                |
| B. Realpath-rewrite each entry to /nix/store form  | Yes             | Partial (looks diff) | none                     | low-med            |
| C. Relax filter to "realpath in /nix/store"        | No ($HOME dep)  | Yes                  | none                     | low                |
| D. B + dedup                                       | Yes             | Partial              | none                     | low-med            |
| E. Verbatim host PATH                              | No (/usr leaks) | Yes                  | n/a                      | high (noise)       |
| F. Fixed profile-dir list, ignore host PATH        | Yes             | Yes                  | none                     | medium (no devshell pickup) |
| G. Lazy `[[ -d ]]` rebuild inside tent             | Yes             | Yes                  | varies                   | low                |

The mount-cost column collapses to zero across the board once Q1's
C+F binds (`/nix/var`, `~/.nix-profile`, `/etc/nix`) land, so the
deciding axis is hermeticity vs. debuggability, not mount footprint.

Soft commitment: **B (realpath-rewrite, no dedup)** for v0. Rationale:

- In-tent PATH doesn't embed `$HOME`, so future subagent / different-
  user-inside-tent contexts (FDR-0001) don't need a PATH rewrite.
- "Looks different" cost is small for a nix user: `/nix/store/<hash>/
  bin` is the canonical form they read fluently outside the tent too.
- Dedup (D) is an optimization, not a correctness change; defer until
  PATH length is shown to actually cost something.
- F is tempting (zero host-shell-hygiene leak) but reimplementing
  nix-develop's PATH computation to pick up the user's devshell is a
  losing game.

Soft, not hard: if a tool inside the tent turns out to introspect its
own location via `/proc/self/exe` *and* keys off the profile-link
path, we'd fall back to A or C for that tool. We don't have a known
case today but the table stays on file.

### Update (2026-05-19, cont'd): in-tent `nix install / run` is a target use case

The C+F + B shape gives the in-tent agent read-only access to the
host's nix profile-link directories — enough for `nix --version`,
`nix flake show`, `nix build`, and `nix shell` / `nix run` against
existing flakes (these go through the daemon, which writes to
/nix/store on the host; the tent reads through the RO bind). The
daemon connection is `Trusted: 0` so cache pushes are denied, but
local builds work.

A desired-but-not-yet-wired capability: **the in-tent agent should be
able to use `nix profile install`** to put a tool into a tent-scoped
profile, then use that tool for the rest of the session. The use case
is "the agent realizes it needs `ripgrep-all` to grep PDFs; it should
be able to install it itself without leaving the tent."

Concrete shape of the question (not committed):

- `nix profile install` writes to a profile path; the host user's
  default profile (`~/.local/state/nix/profiles/profile` or
  `~/.nix-profile`) is RO-bound inside the tent, so installs there
  fail. Don't open that — it pollutes the user's host environment.
- Alternative: clown materializes a per-session tent profile (e.g.
  `$XDG_STATE_HOME/clown/tent-profiles/<session-id>/`) and binds it
  writable; sets `NIX_USER_PROFILE_DIR` so `nix profile` defaults
  there. Profile disappears when the session ends (or accumulates
  until the user `nix-collect-garbage`s).
- Or: rely entirely on `nix shell`/`nix run` for transient tool
  acquisition (no profile writes needed). Cheaper to wire, but
  forces the agent to re-resolve dependencies every subshell.
- Or: pre-populate a tent-wide profile during image build with a
  curated dev floor (the "image baseline expansion" path Q1 rejected,
  resurfacing here as a profile rather than image contents).

Not blocking the C+F+B v0 — the agent can already do useful work
without `nix profile install` — but flagged so the design space stays
visible. Probably its own ADR once the v0 lands and we have
operational data on how often the in-tent agent actually wants to
install something at runtime.

### Update (2026-05-19, cont'd): host ssh-wrapper convention drives bind list

Running the host-side bats smoke (`just test-tent-smoke`) surfaced an
ssh-side analog to the home-manager nix-profile finding: the user's
home-manager ships an ssh wrapper that always prepends:

    -o UserKnownHostsFile=$SSH_HOME/known_hosts -F $SSH_HOME/config

with `SSH_HOME` defaulting to `~/.config/ssh` per the user's
home-manager module. The wrapper is deliberately split from `~/.ssh/`
so the latter (which may hold private key material) stays untouched
by the tent allowlist while the former (public client config +
known_hosts only) is safe to mount.

With only the original C+F mounts active, `ssh -T git@github.com`
inside the tent failed with `Can't open user config file /config`
because:

- `SSH_HOME` was not in DefaultEnvPassthrough → the var was unset
  inside the tent → the wrapper expanded `$SSH_HOME/config` to
  `/config`.
- `~/.config/ssh/` was not in DefaultReadOnlyBindCandidates → even if
  SSH_HOME had pointed somewhere reasonable, the directory would
  have been absent inside the tent namespace.

Fix (committed): add `~/.config/ssh` to `DefaultReadOnlyBindCandidates`
and `SSH_HOME` to `DefaultEnvPassthrough`. Both are gated on host
presence — users without this convention have `~/.config/ssh` absent
and `SSH_HOME` unset, so the additions are no-ops for them.

This is the same shape lesson as the nix-profile case: the agent's
useful work depends on a small, durable set of *host conventions for
how user-facing tools find their config*. The tent allowlist needs to
mirror those conventions, not just the tool-canonical paths. Cases
likely to come next on the same axis: `~/.config/gh` (GitHub CLI),
`~/.config/op` (1Password CLI for auth tokens — note the security
trade-off there), `~/.config/aws`, `~/.npmrc`. Each will be a per-host
decision once it surfaces.

The shell-wrapper detail itself is also a reminder: home-manager
indirection means "what's the wrapper actually doing?" is sometimes
the right debugging question, even when the user-facing binary looks
unwrapped. Two layers up from `~/.nix-profile/bin/ssh` was a
`bash -lc` wrapper injecting flags that depended on env state the
tent hadn't been told to forward.

## Provisional recommendation

If forced to commit today: **standardize on fence** as the
underlying sandbox tech (uniform Linux+macOS, declarative policy
file, clown can compose policies per-provider), and **make tent
mandatory** with FDR-0009's `--naked` as the named opt-out. The
binary-patch path for claude-code becomes defense-in-depth — issue
[#62][i62] (claude-code 2.1.123+ bump) gets unblocked because the
binary's own managed-settings stop being load-bearing for the
unbypassability property.

This is **not** a committed recommendation. The sandbox-tech
choice is its own ADR; this section exists to give the reader a
plausible end state.
