# Hats — Profile-Based Plugin and Fragment Selection

## Abstract

This record sketches the design space for "hats": named profiles that
narrow clown's plugin and system-prompt-fragment surface to a curated
subset for a given session. Status is **exploring** — questions below
are intentionally open and will be answered in future revisions before
this becomes implementable.

## Mental model

A hat is a named profile that narrows clown's plugin +
system-prompt-fragment surface to a curated subset. Useful for
context-window hygiene (only load tools relevant to the task) and intent
signaling ("I'm wearing the security-review hat — don't suggest writing
new features").

## Key design axes

1. **Binding time.** Build-time variants (each hat = a separate
   `mkCircus` call → separate binary) vs runtime selection (one binary
   bakes the superset, a flag/env names which subset to activate at
   startup).
2. **Definition location.** Defined in `mkCircus` (Nix), in user config
   (`~/.config/clown/hats/<name>.toml`), or contributed by a plugin (a
   hat shipped as `.clown-plugin/hats/`).
3. **Composition.** Single-hat-at-a-time, or stackable
   (`--hat security --hat writing` unions both)?
4. **Plugin/fragment coupling.** Does a hat bundle both kinds together,
   or are plugin-hats and fragment-hats orthogonal selectors?

## Provisional recommendation

Runtime selection from a build-time superset, hats defined in user
config, single active hat, bundles both plugins and fragments.
Tradeoff: every plugin's closure stays alive on disk even when
disabled, but the win is "switch hats without a rebuild," which is
essential for daily ergonomics.

## Most load-bearing clarifier — what's driving this?

- **Context-window hygiene** ("the model wastes tokens on tools I'm
  not using")?
- **Intent signaling** ("tell the model what mode I'm in")?
- **Safety** ("no shell or git tools in the planning hat")?
- **All three?**

The motivation pins down whether disabled plugins should be (a) hidden
from the model entirely (their `--plugin-dir` is filtered out at clown
startup), (b) loaded but with tools suppressed via managed-settings deny
patterns, or (c) loaded but flagged in the system prompt as "not active
right now."

## Open questions for a future revision

- Answer the load-bearing clarifier above.
- Pin each of the four design axes.
- Resolve interaction with existing mechanisms: `--plugin-dir`,
  `CLOWN_DISABLE_CLOWN_PROTOCOL`, FDR 0003's prompt-fragment layering,
  managed-settings.
- Decide whether hats become an FDR (single feature design) or warrant
  an RFC (interface contract that plugins implement, e.g. if hats are
  plugin-contributed).
